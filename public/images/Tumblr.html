                  Inheritance & Modules

                  Response to Sandi Metz’s Practical Object-Oriented Design in Ruby, Chapters 6 and 7

                  Inheritance:
                  "It solves the problem of related types that share a great deal of common behavior but differ across some dimension."
                  It is better when creating a subclass to push abstract code up rather than pushing concrete code down. Subclasses have no need to know about the abstractions in the superclass and hook methods allow for this.
                  One clue that the inheritance hierarchy is well-designed is if it is easy to add new subclasses. A technique to allow for other programmers to easily add subclasses is to provide helpful documentation when there is an error, for example if a required method is not included in the new subclass.

                  Modules:
                  Modules allow “objects that play a common role [to] share behavior.” The Liskov Substitution Principle is a mathematical way of saying that “a subtype should be substitutable for its supertype”. Shallow class hierarchies are preferable to deep ones as they are easier for the brain to understand. Narrow hierarchies are better than wide hierarchies but don’t prioritize narrow-ness over shallow-ness because deep hierarchy structures have a tendency to expand both vertically and horizontally.

                      Classes: Single Responsibilities

                      Response to Sandi Metz’s Practical Object-Oriented Design in Ruby, Chapter 2: Designing Classes with a Single Responsibility 

                      My reaction:

                      Creating a class is like fiddling with a gene: It may wither and die or it might just propagate into other code, into new code, and live FOREVER. Be careful and strive to make it coherent, flexible, and focused.

                      The Ideal Class and the Ideal Method:

                      Metz argues that if you must use a conjunction such as “or” or “and” to summarize what a class does, it is probably doing too much. The ideal class consists only of things that are “highly related to its purpose.” Furthermore, there is more room for changes and updates to the program when its classes do just one thing and those things are isolated from each other.

                      Along these lines, methods should not reference another method’s structure because the former method will get disrupted if the latter method’s output changes.

                      Reasons to Set the Bar High:

                      Metz also puts forward the idea that all classes, even ones without dependencies, should be altered to reflect this standard because the code written now not only “reveals […] design intentions” but also because it will serve as a pattern that could be “replicated forever.” Kinda like a gene. You don’t want to make a shoddy job of it.

                      Improve It Now vs. Improve It Later:

                      "[Make] informed tradeoffs between the needs of the present and the possibilities of the future." -Metz

                      Take-Aways:

                      Metz concludes that if you write code using good practices, the design will reveal itself. I concluded not to make any needy cyclist friends.

                            Iteration, Scope, & .each

                            Basic Example:

                            The .each iterator works on each element in the array. It will pass these elements separately through the block of code that follows it.

                            Scope:

                            At all times there is some sort of environment in the program. We can’t have the program do all things at all times. In fact, it may be helpful for the program to forget things. Think of variables as references to something.

                                     def puralize(word)
                                              word + “s”
                                     end

                                     puralize(red)                                                             ==> “reds”

                            Iterating with an Array to Create a New Array:

                            array = [0, 100, 35, 36] 
                            over_35 = []

                            array.each do |number|
                                     if number > 35
                                              over_35 « number
                                     end
                            end                                                                                ==>  [100, 36]

                              Iteration

                              States, Conditions, and Increments

                              Here are two examples of programs that print “hello” to the console three times, one in JavaScript the other in Ruby:

                              JavaScript, a language that is more explicit than Ruby, reads like this:

                              While i is 0, 1, and 2, “hello” will print to the console:

                              State: i = 0
                              Condition: i < 3
                              Increment: i ++ (increases i by one each time)

                              Ruby, a language is a little more like English, reads like this:

                              While i is 0, 1, and 2, “hello” will print to the console:

                              State: i = 0 (implicit)
                              Condition: 3.times
                              Increment: 3.times

                              The interesting thing about Ruby is that the times. operator says, “When i == 3, don’t print hello,” while also incrementing by 1.

                              Conclusions

                              There’s so much going on behind the scenes with Ruby that comparing it with a language like JavaScript helped me understand the three gears of iteration: state, condition(s), and increment. Explicit languages like JavaScript cater to understanding while Ruby caters to elegance.

                                  Fizzbuzz

                                  Here’s my first ever fizzbuzz, written in Ruby:

                                  Description:

                                  • loops 101 times
                                  • checks if i is 0 and prints nothing if it is
                                  • checks if i is not mod 3 or mod 5 and puts i if this is the case
                                  • checks if i is mod 3 and mod 5 and puts fizzbuzz if this is the case
                                  • checks if i is mod 3 and puts fizz if this is the case
                                  • checks if i is mod 5 puts buzz if this is the case

                                  When comparing my solution to others posted online, I found a solution on Ruby Quiz #126:

                                  http://rubyquiz.com/quiz126.html

                                  This solution uses the number one as self so it runs only one-hundred times and starts with one. This is an elegant way to navigate i defaulting to zero. It then first checks for numbers with mod 3 and mod 5, which I found interesting as integers with both mods are infrequent. Upon closer examination, my solution is like Ruby Quizzes’ solution’s shadow-mirror self: it’s a reversed negative, plus a bulky zero-avoidance strategy.

                                          Test-Driven Development

                                          Notes from today’s lecture on Test-Driven Development or TDD

                                          Steps for TTD:

                                          1. Ensure that test is associated with a working file (require “./file”)
                                          2. Run test (rspec filename.filetype)
                                          3. Make sure test fails
                                          4. Write code in working file test to pass in working file
                                          5. Run test
                                          6. Repeat Steps 4 and 5 until test passes
                                          7. Refactor

                                                In response to “Optimism”

                                                In response to: http://braythwayt.com/homoiconic/2009/05/01/optimism.html
